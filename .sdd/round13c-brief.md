## Аналіз (перевір, не приймай на віру)
Токени (accessToken/refreshToken) лежать у localStorage WebView, який у Telegram СПІЛЬНИЙ для всіх акаунтів користувача на одному пристрої. При старті апка бачить валідний збережений токен і пропускає авторизацію по initData — тому другий акаунт отримує сесію першого.

## Задача — СИСТЕМАТИЧНЕ НАЛАГОДЖЕННЯ І ФІКС
1. Вивчи поточний init-флоу AuthProvider: коли викликається POST /api/me/auth з initData, коли береться збережений токен, як зберігається користувач.
2. Фікс-інваріант: **сесія завжди належить актуальному Telegram-користувачу з initData**. Канонічний підхід: при кожному старті, якщо є Telegram initData з user.id — порівняти з ідентичністю збереженої сесії (tg-user-id збережений поруч із токенами, або id з JWT payload якщо він = tg id — перевір що в JWT: там БД-id, тому зберігай tg_user_id окремим ключем при логіні); якщо не збігається — повністю очистити сесію (токени + повʼязані кеші користувача) і пройти авторизацію по initData заново (з онбордингом: умови/нік — для нового користувача бекенд поверне termsAccepted=false і флоу сам поведе).
3. Уважно: dev-режим без Telegram (VITE_DEV_BYPASS_AUTH) не має зламатись — без initData поведінка як зараз.
4. Перевір і суміжне: leaderboard/profile/tickets кеші або стор зі старим користувачем після переключення — стан юзера має повністю перезавантажитись (перевір liveStore/контексти на залишки).
5. Якщо виявиш, що initData взагалі не ре-валідується на бекенді при наявному токені — задокументуй рекомендацію для бекенда (напр. клейм tg_user_id у JWT + перевірка відповідності у /api/me/auth), НЕ реалізовуючи.

## Доведення
- Емуляція в Playwright: підстав window.Telegram.WebApp з initData юзера A → пройди auth (стаб бекенда page.route: /api/me/auth повертає токени A, /api/me — юзера A); перезавантаж з initData юзера B → МАЄ піти новий /api/me/auth з initData B (а не використатись токен A), стор/кеші A очищені, онбординг для B показаний (termsAccepted=false у стабі B).
- Регресія: (а) той самий юзер A повторний вхід — без зайвого ре-логіну (сесія переживає перезапуск як зараз); (б) dev-режим без Telegram — працює як раніше.
- tsc чистий; build зелений; lint не гірше базлайну (8 старих errors).
